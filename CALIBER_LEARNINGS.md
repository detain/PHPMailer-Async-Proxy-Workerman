Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern]** PHPStan config in this fork lives at `phpstan.neon.dist` and scopes only the new async/proxy-protocol code: `paths: [src/Async, src/ProxyProtocol]`, `level: 5`, `treatPhpDocTypesAsCertain: false`, plus an `ignoreErrors` block silencing PHP stream-function `expects resource, ... given` false positives (`fwrite`, `fread`, `feof`, `fgets`, `fclose`, `stream_*`). Run with `vendor/bin/phpstan analyse --configuration phpstan.neon.dist --no-progress --memory-limit=512M`. Wired into CI via a separate `.github/workflows/static-analysis.yml` workflow, not the test matrix.
- **[gotcha]** Watch for duplicated ternary branches — e.g. `$cond ? inet_pton($a).inet_pton($b) : inet_pton($a).inet_pton($b)` where both sides compute the same thing make the condition dead code, and PHPStan flags the downstream `$x === false` check as "Strict comparison using === between string and false will always evaluate to false". The fix is to compute each part once into a local (`$src`, `$dst`) and check each `inet_pton()` result independently before concatenating.
- **[convention]** CI matrix in this fork targets PHP 8.1–8.4 as stable, with 8.5 and `nightly` marked `experimental: true` via `continue-on-error`. Two test jobs run in parallel: `test-sync` (full upstream suite + sync transport, installs postfix + smtp-sink for legacy `SendTestCase` tests) and `test-async` (just `test/Async/` + `test/ProxyProtocol/`, no mail server needed). - **[gotcha]** You cannot pass a `protected` method as `[$this, 'methodName']` to a strictly-typed `?callable` parameter declared in another namespace — PHP's strict-callable check rejects it with `TypeError: Argument #1 ($handler) must be of type ?callable, array given`. This bit `SMTP::setTransport()` when handing `[$this, 'errorHandler']` (protected) to `Transport::setErrorHandler(?callable)`. Fix: wrap in a `\Closure` bound to `$this`. The array-callable form only works for `public` methods or when the caller is in the same class.
- **[gotcha]** `stream_socket_enable_crypto($sock, true, $method)` on a non-blocking stream returns THREE distinct values, not just bool: `true` = handshake complete, `false` = handshake failed (cert/protocol mismatch), and **integer `0` = would-block, kernel needs more I/O — caller MUST retry**. The strict-equality check `=== 0` is required; `!$ok` would treat would-block as failure. **Important nuance discovered after first wiring it:** the naïve "suspend on `onReadable` OR `onWritable`" pattern busy-loops because a connected TCP socket with send-buffer space is always writable, so `onWritable` fires immediately every iteration. Use `onReadable` + a small **exponential-backoff timer (1ms → 50ms cap)** instead — `onReadable` covers the realistic blocker (server-hello / finished arriving) and the timer covers the rare "handshake wants more write room" case. See `WorkermanTransport::cryptoLoop()`.
- **[gotcha]** Self-signed `tls://127.0.0.1:0` listener + `pcntl_fork` + `stream_socket_accept` is FLAKY as a positive-path TLS test — the child often exits with no accepted connection inside the timing window. Prefer testing the *would-block loop* itself: connect to a silent plain-TCP listener, call `enableCrypto()` with a 2s budget, and assert the call sat ~1.5–4s (proving it suspended) and returned false. Real TLS handshake success is best verified against a live SMTPS test server in `MailTransportTest`, not a fork-and-accept in-process fixture.

- **[gotcha]** `.gitignore` uses `docs/*` followed by explicit allowlist lines (`!docs/README.md`, `!docs/UPSTREAM.md`). Adding any new committed file under `docs/` requires another `!docs/<FILE>` line — otherwise `git add docs/NEWFILE.md` silently does nothing and the next `git status` shows clean even though the file exists on disk. Same trap applies to `.claude` / `.caliber` etc. which are ignored at the dotfile level.
- **[gotcha]** PHPMailer (upstream) prepends `ssl://` (and sometimes `tls://`) onto `$host` when `SMTPSecure='ssl'`, so `stream_socket_client()` performs an implicit TLS handshake **before returning** — which means any PROXY-protocol header would land AFTER the TLS ClientHello and port-465 receivers reject it. Both `StreamTransport::connect()` and `WorkermanTransport::connect()` work around this by: regex-stripping the scheme, connecting as plain `tcp://`, writing the PROXY bytes, then upgrading via `stream_socket_enable_crypto()` (or `cryptoLoop()` for the async path). Mirror that pattern in any new Transport implementation; see `test/Async/ProxyBeforeImplicitTlsTest.php` for the regression test.
- **[gotcha]** `SMTP::getSMTPConnection()` must return a TRUTHY value when the transport has no PHP stream resource (Workerman / coroutine / any wrapper around an OS-level socket that doesn't surface as a userland `resource`). Returning `false` because `$transport->getResource() === null` makes `SMTP::connect()`'s `=== false` check kill the session before the first protocol read. The fix is `return $resource ?? true;` — and `SMTP::connected()` should call `$transport->isOpen()` instead of `is_resource($this->smtp_conn)`. See `test/Async/StreamlessTransportTest.php`.
- **[pattern]** When `FiberRunner` (or any helper that runs a private Revolt loop) is called from code that has *already* registered watchers / deferreds on the global driver but has NOT yet started the reactor, create a brand-new driver via `(new EventLoop\DriverFactory())->create()`, `EventLoop::setDriver($isolated)` for the duration of the run, and restore the previous driver in `finally`. Otherwise `$driver->run()` ticks the caller's pending callbacks (and our `$driver->stop()` kills them when our fiber finishes). See `FiberRunner::runWithPrivateLoop()` and the `testPrivateLoopDoesNotFireCallersPreScheduledCallbacks` regression test.
- **[pattern]** Setting GitHub repo metadata after a docs merge: `unset GITHUB_TOKEN; gh repo edit OWNER/REPO --description "..." --homepage "..." --add-topic foo --add-topic bar ...` — repeat `--add-topic` per topic in a single call. Verify with `gh repo view OWNER/REPO --json description,homepageUrl,repositoryTopics`. Topics added via `--add-topic` are sorted alphabetically server-side, not preserved in insertion order.
- **[convention]** Auto-detect (`V1Header::auto()` / `Configurator::v1()`) now THROWS when the source IP is neither valid IPv4 nor IPv6 rather than silently degrading to `PROXY UNKNOWN\r\n`. - **[gotcha]** `SMTP::connected()` must preserve upstream's three-state contract: EOF on a previously-open socket should emit the `SMTP NOTICE: EOF caught while checking if connected` debug line AND call `$this->close()` as a side effect, not just return false. The fork's transport-based rewrite originally dropped this because it short-circuited on `!$transport->isOpen()` before checking metadata. Correct shape: check `$transport->getMetadata()['eof']` FIRST, and gate the notice+close on whether there ever was a connection (`getResource() !== null || $this->smtp_conn`), then return `$transport->isOpen()` as the fallback.

- **[gotcha]** A transport built on top of Workerman's `AsyncTcpConnection` (rather than raw `stream_socket_client` + Revolt) CANNOT wrap each `Transport` method (`connect` / `readLine` / `write` / `close`) in its own `FiberRunner::run(...)` call. `AsyncTcpConnection` registers long-lived `onMessage` / `onClose` watchers in `connect()`; those watchers belong to whatever Revolt driver was active at registration time, and `FiberRunner::run()` swaps in a fresh private driver per invocation — so subsequent `readLine()` calls fiber-suspend on a driver that never receives the readable events and the test hangs forever (eats the PHPUnit 30s default and looks like a deadlock). The contract for any AsyncTcpConnection-backed transport is: caller wraps the WHOLE SMTP session in **one** `FiberRunner::run()`, the transport methods just `EventLoop::getSuspension()` directly. Inside a real running Workerman worker the loop is always alive so this is free.
- **[gotcha]** Workerman's Revolt-driver adapter file is `vendor/workerman/workerman/src/Events/Fiber.php` (`Workerman\Events\Fiber`), NOT `Events/Revolt.php` — the class name says `Fiber` because it relies on PHP Fibers, but it IS the Revolt bridge (`__construct()` does `$this->driver = EventLoop::getDriver()`). Use it as the global Workerman event-loop adapter when bootstrapping AsyncTcpConnection outside a real Worker (PHPUnit / CLI): `Worker::$globalEvent = new \Workerman\Events\Fiber()`. Re-bind it whenever the current Revolt driver changes (each `FiberRunner` private loop creates a fresh driver) — cache the driver you bound to and refresh when it differs.
- **[gotcha]** When a test fake subclasses `SMTP` and tracks call counts for `close()`, make the override **idempotent** (early-return when already closed). The connection pool's `safeClose()` calls `quit()` (which internally closes on success) and then `close()` again as belt-and-suspenders — a naive `closeCount++` on every entry will double-count and break assertions. Mirror the real `SMTP::close()` shape: guard the body on `$this->connected` / equivalent state and flip the flag at the top.
- **[pattern]** When a PHPUnit test (especially anything touching `AsyncTcpConnection` / Revolt / fiber-suspension) hangs and `timeout 30 vendor/bin/phpunit` produces an empty `{}` failure result, drop down to a standalone `php /tmp/diag-foo.php` script that replicates ONLY the transport + a `pcntl_fork`'d listener — no PHPUnit. The diag prints stepwise progress (`Connecting...`, `Connect returned: true`, `Trying readLine...`) so you can see which call deadlocks and against which event-loop driver (`get_class(EventLoop::getDriver())`). Running the full test runner masks where the suspension is parked.
- **[gotcha]** In this Claude harness `BashOutput` / `KillShell` do not exist — the deferred tools for managing `run_in_background: true` bash tasks are `TaskOutput` (with `block: true, timeout: <ms>`) and `TaskStop`. Output for a background task is also tail-able directly from `/tmp/claude-1000/<session>/tasks/<task_id>.output`. `ToolSearch` with `select:BashOutput` returns no matches — use a keyword query like `read background task output` to find the right tool names.
- **[gotcha]** PHPStan 1.x flags `private` properties that are assigned but never read (e.g. `errorSink` kept for API-symmetry with sibling transports) under rule `property.onlyWritten`. Silence with a `@phpstan-ignore-next-line property.onlyWritten` doc comment immediately above the property — adding it to the class-level `ignoreErrors` in `phpstan.neon.dist` is overkill for a one-off. Don't delete the property just to placate the linter when other transports in the family read it.
