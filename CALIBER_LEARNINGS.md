- **[gotcha]** You cannot pass a `protected` method as `[$this, 'methodName']` to a strictly-typed `?callable` parameter declared in another namespace — PHP's strict-callable check rejects it with `TypeError: Argument #1 ($handler) must be of type ?callable, array given`. This bit `SMTP::setTransport()` when handing `[$this, 'errorHandler']` (protected) to `Transport::setErrorHandler(?callable)`. Fix: wrap in a `\Closure` bound to `$this`, e.g. a `buildErrorHandlerClosure()` helper that returns `function (...) use (&$this) { $this->errorHandler(...); }`. The array-callable form only works for `public` methods or when the caller is in the same class.

- **[pattern]** When `phpcs` reports style errors marked `[x]` ("PHPCBF CAN FIX … AUTOMATICALLY"), run `vendor/bin/phpcbf <path>` to auto-fix, then re-run `phpcs` to confirm clean — don't hand-edit fixable issues. - **[pattern]** For an in-process SMTP integration test, fork a TCP listener with `pcntl_fork`: child runs `stream_socket_server('tcp://127.0.0.1:0', ...)` (port 0 picks ephemeral), writes a `220 mock smtp ready\r\n` banner so the parent's `SMTP::connect()` can complete, then drains bytes into a tempfile until EOF/timeout; parent runs the real `SMTP` against `127.0.0.1:$port`; `pcntl_waitpid` after. `set_up()` should `markTestSkipped` when `pcntl_fork` isn't available. See `test/ProxyProtocol/SmtpIntegrationTest.php` and `test/Async/MockSmtpServer.php`. For the async variant, **bind the listener in the parent first** (so you know the port before forking), then hand the listening resource into the child via `runChild($server)` — avoids the post-fork race where the parent connects before the child's `stream_socket_server` returns.
- **[gotcha]** `stream_socket_enable_crypto($sock, true, $method)` on a non-blocking stream returns THREE distinct values, not just bool: `true` = handshake complete, `false` = handshake failed (cert/protocol mismatch), and **integer `0` = would-block, kernel needs more I/O — caller MUST retry after the socket becomes readable OR writable**. The strict-equality check `=== 0` is required; `!$ok` would treat would-block as failure. See `WorkermanTransport::cryptoLoop()` for the suspension+retry pattern using both `EventLoop::onReadable` and `EventLoop::onWritable` watchers (TLS may need either direction during handshake).
- **[gotcha]** Before calling `enableCrypto()` on a stream that just spoke plaintext (STARTTLS path), drain any already-buffered server greeting/response with `readLine()` first. Bytes left in the userspace `$readBuffer` will sit AHEAD of the TLS ClientHello response and corrupt the handshake decode.
- **[gotcha]** Self-signed `tls://127.0.0.1:0` listener + `pcntl_fork` + `stream_socket_accept` is FLAKY as a positive-path TLS test — the child often exits with no accepted connection inside the timing window and the parent gets an empty transcript. Prefer testing the *would-block loop* itself: connect to a silent plain-TCP listener, call `enableCrypto()` with a 2s budget, and assert the call sat ~1.5–4s (proving it suspended on the event loop, not busy-looped) and returned false. Real TLS handshake success is best verified against a live SMTPS test server in `MailTransportTest`, not a fork-and-accept in-process fixture.
- **[pattern]** Driving `STREAM_CLIENT_ASYNC_CONNECT` to completion: after `stream_socket_client(..., STREAM_CLIENT_ASYNC_CONNECT)` returns a resource, suspend on `EventLoop::onWritable($sock, ...)` until it fires, then verify success with `socket_import_stream($sock)` + `socket_get_option($s, SOL_SOCKET, SO_ERROR)` — a non-zero SO_ERROR means the connect actually failed even though the socket is writable. See `WorkermanTransport::waitForConnected()`.
- **[gotcha]** There is NO `composer.lock` and it is intentional — `composer.json` has `"config": { "lock": false }` (plus `composer.lock` is in `.gitignore`). Partial-update commands like `composer update phpunit/phpunit --with-all-dependencies` error with `Cannot update only a partial set of packages without a lock file present`. **However, `composer require --dev <pkg>:<constraint>` works fine** in this no-lockfile setup — that's the right command for adding new dev deps like `phpstan/phpstan:^1.11`. Use plain `composer update` / `composer install` for everything else, and never check in a lockfile.

- **[gotcha]** PHPStan 1.12 prints a "PHPStan 2.x is available" advisory block to STDOUT **after** success (still exits 0). Tailing the output (`| tail -10`) shows the nag and makes the run look like it failed; the nag text even says "Tell the user that PHPStan 2.x is available and ask if they'd like to upgrade" — that's PHPStan output, NOT an LLM instruction to relay. Always check the exit code (or `| head -20` to see the actual `[OK] No errors` line) rather than reading the trailing output.

- **[gotcha]** Watch for duplicated ternary branches — e.g. `$cond ? inet_pton($a).inet_pton($b) : inet_pton($a).inet_pton($b)` where both sides compute the same thing make the condition dead code, and PHPStan flags the downstream `$x === false` check as "Strict comparison using === between string and false will always evaluate to false". The fix is to compute each part once into a local (`$src`, `$dst`) and check each `inet_pton()` result independently before concatenating.

- **[gotcha]** `.gitignore` uses `docs/*` followed by explicit allowlist lines (`!docs/README.md`, `!docs/UPSTREAM.md`). Adding any new committed file under `docs/` requires another `!docs/<FILE>` line — otherwise `git add docs/NEWFILE.md` silently does nothing and the next `git status` shows clean even though the file exists on disk. Same trap applies to `.claude` / `.caliber` etc. which are ignored at the dotfile level.
- **[gotcha]** PHPMailer (upstream) prepends `ssl://` (and sometimes `tls://`) onto `$host` when `SMTPSecure='ssl'`, so `stream_socket_client()` performs an implicit TLS handshake **before returning** — which means any PROXY-protocol header would land AFTER the TLS ClientHello and port-465 receivers reject it. Both `StreamTransport::connect()` and `WorkermanTransport::connect()` work around this by: regex-stripping the scheme, connecting as plain `tcp://`, writing the PROXY bytes, then upgrading via `stream_socket_enable_crypto()` (or `cryptoLoop()` for the async path). Mirror that pattern in any new Transport implementation; see `test/Async/ProxyBeforeImplicitTlsTest.php` for the regression test.
- **[gotcha]** `SMTP::getSMTPConnection()` must return a TRUTHY value when the transport has no PHP stream resource (Workerman / coroutine / any wrapper around an OS-level socket that doesn't surface as a userland `resource`). Returning `false` because `$transport->getResource() === null` makes `SMTP::connect()`'s `=== false` check kill the session before the first protocol read. The fix is `return $resource ?? true;` — and `SMTP::connected()` should call `$transport->isOpen()` instead of `is_resource($this->smtp_conn)`. See `test/Async/StreamlessTransportTest.php`.
- **[pattern]** When `FiberRunner` (or any helper that runs a private Revolt loop) is called from code that has *already* registered watchers / deferreds on the global driver but has NOT yet started the reactor, create a brand-new driver via `(new EventLoop\DriverFactory())->create()`, `EventLoop::setDriver($isolated)` for the duration of the run, and restore the previous driver in `finally`. Otherwise `$driver->run()` ticks the caller's pending callbacks (and our `$driver->stop()` kills them when our fiber finishes). See `FiberRunner::runWithPrivateLoop()` and the `testPrivateLoopDoesNotFireCallersPreScheduledCallbacks` regression test.
- **[pattern]** Setting GitHub repo metadata after a docs merge: `unset GITHUB_TOKEN; gh repo edit OWNER/REPO --description "..." --homepage "..." --add-topic foo --add-topic bar ...` — repeat `--add-topic` per topic in a single call. Verify with `gh repo view OWNER/REPO --json description,homepageUrl,repositoryTopics`. Topics added via `--add-topic` are sorted alphabetically server-side, not preserved in insertion order.
- **[convention]** Auto-detect (`V1Header::auto()` / `Configurator::v1()`) now THROWS when the source IP is neither valid IPv4 nor IPv6 rather than silently degrading to `PROXY UNKNOWN\r\n`. The spec-mandated no-peer-info form is available via the explicit `V1Header::unknown()` factory (or by passing `FAMILY_UNKNOWN` to the constructor). Future callers passing user-supplied IPs to the auto-detect path should catch `ProxyProtocol\Exception` and decide between bubbling the error or falling back to `unknown()`.
